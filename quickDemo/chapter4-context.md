## 1. ReAct (Reasoning + Acting)

**ReAct = Reasoning + Acting**，[Yao et al. 2022](https://arxiv.org/abs/2210.03629)

核心思想：模仿人类解决问题的方式，将**推理 (Reasoning)** 与**行动 (Acting)** 显式结合，形成"思考-行动-观察"循环。

```
Thought: 分析当前情况、分解任务、规划下一步
Action:  调用外部工具（如 Search['query'] 或 Finish[最终答案]）
Observation: 工具返回结果
... 重复 ...
Thought: 已获得足够信息
Action:  Finish[最终答案]
```

形式化表达：
- 在时间步 t，策略 π 根据问题 q 和历史轨迹生成思考和行动：`(th_t, a_t) = π(q, (a_1,o_1),...,(a_{t-1},o_{t-1}))`
- 工具 T 执行行动并返回观察：`o_t = T(a_t)`

核心要素：
- **Thought (思考)**：智能体的"内心独白"，分析情况、分解任务、反思结果
- **Action (行动)**：决定采取的具体动作，通常调用外部工具
- **Observation (观察)**：执行 Action 后从工具返回的结果

与纯推理（CoT）和纯行动的区别：**思考与行动相辅相成**。思考指导行动，行动结果反修正思考。

特点：
1. **高可解释性**：通过 Thought 链可清晰看到每步推理过程
2. **动态规划与纠错**：根据 Observation 动态调整后续策略（走一步看一步）
3. **工具协同**：LLM 负责推理，工具负责执行，突破单一 LLM 局限

局限性：
1. **强依赖 LLM 能力**：推理、指令遵循、格式化输出能力不足时易中断
2. **执行效率问题**：多步串行调用，延迟和成本较高
3. **提示词脆弱性**：微小变动可能影响 LLM 行为
4. **可能陷入局部最优**：步进式决策缺乏全局规划

适用场景：需要外部知识的任务、需要精确计算的任务、需要与 API 交互的任务

> 调试技巧：打印完整提示词、分析原始输出、验证工具输入输出、添加 Few-shot 示例、调整模型或 temperature

## 2. Plan-and-Solve

**Plan-and-Solve = 先规划，后执行**，[Wang et al. 2023](https://arxiv.org/abs/2305.04091)

核心动机：解决思维链 (CoT) 在处理多步骤复杂问题时容易"偏离轨道"的问题。

两阶段工作流：
```
Phase 1: 规划阶段 (Planning)
  → 接收完整问题 → 分解为步骤 → 生成行动计划 P = (p_1, p_2, ..., p_n)
  → P = π_plan(q)

Phase 2: 执行阶段 (Solving)
  → 严格按照计划 → 逐一执行 → 每步依赖原始问题、完整计划和历史结果
  → s_i = π_solve(q, P, (s_1, ..., s_{i-1}))
  → 最终答案 = s_n
```

与 ReAct 的对比：
- ReAct 像侦探：根据蛛丝马迹一步步推理，随时调整方向
- Plan-and-Solve 像建筑师：先绘制完整蓝图，再严格按蓝图施工

核心设计：
- **规划器 (Planner)**：将问题分解为结构化步骤列表，输出 JSON 数组
- **执行器 (Executor)**：逐步执行计划，维护历史状态，将前步结果作为后步上下文
- **智能体 (PlanAndSolveAgent)**：协调器，组合 Planner + Executor

特点：
1. **结构性强**：完整计划在执行前确定，目标一致性高
2. **稳定性好**：步骤间信息传递明确，不易迷失方向
3. **适合内部推理**：不依赖外部工具，纯 LLM 推理链

局限性：
1. **静态计划**：执行中发现某步不可完成时缺乏动态重规划机制
2. **规划质量依赖**：初始计划质量直接决定最终结果

适用场景：多步数学应用题、需要整合多信息源的报告撰写、代码生成任务

## 3. Reflection

**Reflection = 执行 → 反思 → 优化**，[Shinn et al. 2023 (Reflexion)](https://arxiv.org/abs/2303.11366)

核心思想：为智能体引入事后 (post-hoc) 自我校正循环，像人类一样审视工作、发现不足、迭代优化。

三步循环：
```
1. 执行 (Execution): 生成初步解决方案（"初稿" O_0）
2. 反思 (Reflection): 评审员审视初稿，从事实性、逻辑性、效率、遗漏等维度评估
   → F_i = π_reflect(Task, O_i)
3. 优化 (Refinement): 根据反馈修正初稿，生成"修订稿"
   → O_{i+1} = π_refine(Task, O_i, F_i)
... 重复直到反馈为"无需改进"或达到迭代上限 ...
```

与 ReAct/Plan-and-Solve 的区别：
- ReAct/Plan-and-Solve 完成任务后流程即结束
- Reflection 引入内部纠错回路，修正更高层次的逻辑和策略错误
- 将一次性执行转变为持续优化过程

核心设计：
- **记忆模块 (Memory)**：存储"执行-反思"完整轨迹，序列化为上下文
- **三组提示词**：初始执行 Prompt、反思 Prompt（评审员角色）、优化 Prompt
- **终止条件**：反馈包含"无需改进"或达到最大迭代次数

价值：
- 提供内部纠错回路，不依赖外部工具反馈
- 构建"短期记忆"，记录从初稿到最终版本的完整轨迹
- 显著提升复杂任务的最终成功率和答案质量

成本：
- **模型调用开销增加**：每轮至少额外两次 LLM 调用
- **任务延迟显著提高**：串行过程，不适合实时性要求高的场景
- **提示工程复杂度上升**：需为执行、反思、优化分别设计提示词

适用场景：对结果质量/准确性/可靠性有极高要求，且对实时性要求宽松的场景
（如关键业务代码生成、科学推演、决策支持系统）

> **三种范式选择策略**：需要外部工具/动态探索 → ReAct；逻辑确定/多步推理 → Plan-and-Solve；质量优先/可容忍延迟 → Reflection
